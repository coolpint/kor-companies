# 해외 언론 한국 기업 모니터링 계획

## 목표

- 하루 2회 해외 언론에서 한국 주요 기업이 언급된 기사를 점검한다.
- "모든 기사 수집"이 아니라 "주요 기사 포착"을 목표로 한다.
- 한국 언론은 제외하고, 해외 주요 교역국 언론 중심으로 운영한다.
- 기업 목록과 국가/언론 목록은 별도 문서로 분리해 수정 가능하게 유지한다.

## 운영 원칙

1. 안정성을 우선한다.
2. RSS, 공개 피드, 정적 섹션 페이지처럼 구조가 덜 바뀌는 소스를 우선한다.
3. 국가별로 소수의 신뢰 가능한 대형 매체부터 시작하고, 성능이 검증된 뒤에만 확장한다.
4. 한 소스가 실패해도 전체 실행이 실패하지 않도록 부분 성공 구조로 설계한다.
5. 회사명 키워드 매칭은 영문, 국문, 약칭, 대표 브랜드명을 함께 관리한다.

## 범위

### 포함

- 해외 언론 기사
- 한국 주요 기업 및 대표 계열사
- 미국, 중국, 일본, 베트남, 독일, 영국 우선
- 향후 공식 교역 통계에 따라 국가 우선순위 조정

### 제외

- 한국 언론 원문
- SNS 게시물, 포럼, 블로그
- 수집 안정성이 낮은 무한 스크롤 검색 결과 페이지
- 전량 수집이 필요한 장기 아카이빙 목적

## 권장 산출물

각 실행마다 하나의 요약 리포트를 만든다.

- 실행 시각
- 정상 수집된 소스 / 실패한 소스
- 새로 잡힌 기사 목록
- 기사별 회사명, 국가, 매체, 발행 시각, 제목, URL
- 간단한 한글 요약 1~2문장
- 중복 또는 재보도 판단 결과

## 추천 운영 시각

- 1차: 오전 08:00 KST
- 2차: 오후 18:00 KST

이유:

- 오전 실행은 전일 야간과 북미/유럽발 기사를 커버하기 좋다.
- 오후 실행은 아시아/유럽 근무시간대 기사와 당일 추가 이슈를 반영하기 좋다.

## 문서 구조

- `docs/monitoring-plan.md`: 전체 운영 계획
- `docs/watchlists/korean-companies.md`: 감시 대상 기업 기준과 목록
- `docs/watchlists/trade-country-priority.md`: 교역국 우선순위와 국가별 소스 후보

초기에는 문서로 운영하고, 구현 단계에서는 같은 구조를 그대로 `config/*.yml`로 옮기는 것이 좋다.

## 수집 로직 초안

### 1. 입력

- 기업 목록
- 회사별 별칭 목록
- 국가 우선순위
- 국가별 허용 매체 목록
- 제외 도메인 목록

### 2. 수집

- 국가별 허용 소스의 RSS 또는 섹션 페이지를 순회한다.
- 기사 메타데이터만 먼저 수집한다.
- 타임아웃, 재시도, 사용자 에이전트, 마지막 성공 시각을 소스별로 관리한다.

### 3. 필터링

- 기사 제목과 요약에서 회사명/별칭 키워드를 매칭한다.
- 허용 도메인에 없으면 버린다.
- 한국 언론 도메인이면 버린다.
- 너무 오래된 기사는 버린다.

### 4. 정규화

- URL canonicalization
- 제목 정규화
- 발행 시각 표준화
- 언어, 국가, 매체 메타데이터 부여

### 5. 중복 제거

- canonical URL 기준 1차 중복 제거
- 정규화한 제목 + 발행 시각 근접도 기준 2차 중복 제거
- 동일 이슈의 재배포 기사에는 묶음 ID 부여

### 6. 점수화

기사 우선순위를 간단히 매긴다.

- 매체 신뢰도
- 제목에 회사명이 직접 등장하는지
- 기사 발행 시각이 최근인지
- 사업/실적/규제/공급망/대형 계약 같은 핵심 키워드 포함 여부

### 7. 출력

- 점수가 높은 기사부터 정렬
- 같은 회사 기사 묶기
- 최종 요약 리포트 생성

## 안정성 설계

### 소스 계층화

- Tier 1: 공식 RSS 또는 피드가 있는 대형 매체
- Tier 2: 구조가 단순한 섹션/태그 페이지
- Tier 3: 검색형 또는 보조 소스

초기 MVP는 Tier 1만 사용한다.

### 실패 허용

- 소스별 실패는 경고로 기록하고 전체 실행은 계속한다.
- 실패율이 일정 기준을 넘을 때만 운영 알림을 보낸다.

### 변경 감지

- 각 소스의 최근 성공 시각, 최근 항목 수, 파싱 실패 수를 저장한다.
- 갑자기 항목 수가 0이 되면 소스 구조 변경 가능성으로 표시한다.

### 재현 가능성

- 원본 피드 항목을 짧은 기간 로컬 DB 또는 JSONL로 보관한다.
- 최종 결과만 보지 말고, 원본 입력도 함께 남겨 디버깅 가능하게 한다.

## 구현 단계 제안

### 단계 1. 운영 기준 확정

- 기업 선정 기준 확정
- 국가 우선순위 기준 확정
- 국가별 초기 소스 1~2개 선정

### 단계 2. MVP

- 문서 기준으로 수집 대상 정의
- RSS 기반 수집기 구현
- 회사명 매칭
- 중복 제거
- Markdown 리포트 생성

### 단계 3. 운영 안정화

- 실패 로그
- 소스 건강 상태 체크
- 재시도/백오프
- 알림 채널 연결

### 단계 4. 품질 개선

- 기사 중요도 점수 조정
- 회사별 별칭 확장
- 국가/언론 추가

## 바로 다음 작업

1. 기업 문서에서 초기 시드 목록을 확정한다.
2. 국가/언론 문서에서 국가별 1차 소스를 확정한다.
3. 구현 시 사용할 설정 형식 YAML 스키마를 정한다.
4. 그다음 RSS 수집 MVP를 만든다.
